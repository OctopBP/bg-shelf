отслеживания дополнений. Задачи отсортированы по фазам: от блокеров до мелочей.
Каждая задача = что делаем, зачем, и критерий готовности (DoD).

## Фаза 1. Безопасность (блокеры расширения)

### 1.1. Закрыть запись в глобальный каталог игр
Сейчас `games`, `game_names`, `contributors`, `game_contributors`, `game_links`
доступны на insert/update/delete любому `authenticated` — вектор вандализма.
- [x] **S-1** Миграция `20260622150000_phase1_security.sql`: для каталожных
  таблиц `select` остаётся всем authenticated; `insert/update/delete` — только
  `is_admin(auth.uid())` (service_role обходит RLS). Применено на remote.
- [x] **S-2** Импорт-скрипт в этом репозитории отсутствует (`scripts/import-games.mjs`
  живёт в `bg-preorders-scraper` и пишет через `service_role`, который полностью
  обходит RLS) — от пользовательских политик не зависит, менять нечего.
- [x] **S-3** `addGameToCollection` пишет не напрямую в `games`, а через
  `SECURITY DEFINER` RPC `cache_game(...)` (insert + `on conflict do nothing`,
  без перезаписи каталога). Мок-хендлер добавлен; добавление в коллекцию работает.

**DoD:** обычный пользователь не может изменить/удалить запись каталога; добавление
игры в коллекцию по-прежнему работает.

### 1.2. Сузить утечки в `profiles` и публичные политики
- [x] **S-4** `profiles`: колоночные привилегии — `grant select (id, username)`
  и `grant update (username)`. `role` скрыта от обычных пользователей. Заодно
  закрыта эскалация привилегий: прежняя update-политика разрешала менять любую
  колонку своей строки (включая `role`) — теперь только `username`.
- [x] **S-5** Публичное (anon) чтение `preorders`/`publishers` подтверждено как
  намеренное (публичный каталог): оставлено и задокументировано в `comment on
  policy …` + `docs/database.md` §4. Живо проверено: anon GET → 200.

**DoD:** `role` не виден обычным пользователям; решение по публичности предзаказов
зафиксировано в комментарии к политике.

### 1.3. Базовая валидация входов API
- [x] **S-6** Хелпер `parseBody(schema, request)` (`src/lib/api/validation.ts`)
  возвращает 400 при невалидном JSON/несоответствии схеме. Применён ко всем
  роутам с телом: `collection` (POST/PUT/PATCH/DELETE), `collections` (POST),
  `collections/[id]` (PATCH), `collections/[id]/members` (POST/DELETE),
  `friends` (POST/PATCH/DELETE), `profile` (PATCH), `command` (POST).
  (`photo` — multipart, валидирует тип/размер файла отдельно.)

**DoD:** каждый POST/PATCH-роут валидирует тело через zod.

## Фаза 2. Производительность и масштабирование

### 2.1. Счётчики игр на стороне БД
- [x] **P-1** Вью `collection_item_counts` (миграция `20260622160000`,
  `security_invoker = on`) с агрегатом `count`. `listCollections` /
  `listCollectionsByOwner` берут счётчики из неё (хелпер `collectionGameCounts`),
  больше не выбирают все `collection_items`. Применено на remote.
- [x] **P-2** `max_rows = 1000` оставлен. Счётчики агрегируются в БД (≤ числа
  коллекций строк), списки игр пагинируются (≤ 201 строки на запрос), agent берёт
  ≤ 200 — ни один запрос не приближается к 1000, тихого обрезания нет.
  Зафиксировано в `docs/database.md` §5.

**DoD:** загрузка списка коллекций не зависит от общего числа игр; счётчик верен
при >1000 игр.

### 2.2. Пагинация
- [x] **P-3** Курсорная пагинация `listCollection` / `listAllGames` (курсор —
  пара `added_at`+`id`, сортировка `desc, desc`, фильтр через `.or(...)`).
  Роут `/api/collection` принимает `cursor`/`limit` и отдаёт `nextCursor`;
  клиент `CollectionApp` догружает страницы (IntersectionObserver + кнопка
  «Показать ещё»).
- [x] **P-4** Переиспользуемый помощник курсора `src/lib/pagination.ts`
  (`encodeCursor`/`decodeCursor`/`clampLimit`, типы `Cursor`/`Page<T>`,
  `DEFAULT_PAGE_SIZE`/`MAX_PAGE_SIZE`) — контракт для будущих списков (лента,
  предзаказы).

**DoD:** списки игр грузятся страницами; есть переиспользуемый помощник курсора.

### 2.3. Меньше вызовов `auth.getUser()`
- [x] **P-5** `userId` пробрасывается из роутов: `listCollections(supabase,
  userId)`, `listAllGames(supabase, userId, …)` — внутренний `auth.getUser()`
  убран. На запрос — один `getUser` (в роуте).

**DoD:** на один запрос — один `getUser`.

### 2.4. Стоимость и скорость LLM-пайплайна
- [x] **P-6** `parseAddCommand` (`resolve.ts`) переведён на `claude-haiku-4-5` —
  это классификация со структурированным выходом, Haiku справляется и кратно
  дешевле/быстрее.
- [x] **P-7** Agent-цикл (`agent.ts`) оставлен на `claude-opus-4-8` (многошаговое
  рассуждение; даунгрейд рассуждающего агента — решение пользователя, а в плане —
  лишь «рассмотреть»), но удешевлён: `output_config.effort = "low"`,
  `MAX_ITERATIONS` 16 → 8; верхний предел (шаги/effort/maxDuration=120с)
  задокументирован в коде и в `docs/database.md` §5.

**DoD:** типовая команда дешевле и быстрее; задокументирован верхний предел
стоимости/времени.

## Фаза 3. Архитектура data-слоя

### 3.1. Единая конвенция доступа к данным
- [x] **A-1** Сверились с гайдом Next 16
  (`01-app/01-getting-started/07-mutating-data.md`) по Server Actions — итог в
  `docs/architecture.md` §1.
- [x] **A-2** Канон — **route handlers** (`/api/**`), не Server Actions:
  уже унифицированный паттерн (createClient → getUser → `parseBody` → lib →
  json), задокументирован в `docs/architecture.md` §1.
- [x] **A-3** «Толстый POST» относился к сценарию миграции на Server Actions;
  раз не мигрируем — устранять нечего. Зафиксировано правило «метод = операция,
  один ресурс — один роут», `/api/collection` ему соответствует.

**DoD:** есть один задокументированный паттерн; новые эндпоинты не плодят
разнородные обёртки.

### 3.2. Разбить мега-компонент
- [x] **A-4** Загрузка данных вынесена в переиспользуемый хук
  `src/hooks/useCollectionData.ts` (коллекции, активная вкладка, игры,
  курсорная пагинация). Диалоги уже были отдельными компонентами. `CollectionApp`
  больше не владеет загрузкой — `docs/architecture.md` §2.
  (Дальнейшее выделение `GameGrid` — низкорисковый follow-up, не делалось без
  возможности runtime-проверки.)

**DoD:** ни один компонент не делает «всё»; логика загрузки переиспользуема.

### 3.3. Стратегия mock-режима
- [x] **A-5** Решение: канон локальной разработки/демо — **seed локального
  Supabase** (`supabase/seed.sql` + `config.toml [db.seed]`). MSW-мок помечен
  deprecated (заморожен, без новых хендлеров), удаление — после валидации
  seed-пути локально с Docker. Подробно — `docs/architecture.md` §3.
  Примечание: seed-инфраструктура подготовлена, но не прогонялась (в этом
  окружении нет Docker) — требует `supabase db reset` на машине разработчика.

**DoD:** решение принято и записано; нет скрытого удвоения работы на каждую фичу.
